# PPT 生成 Skill

## 任务本质

通过生成一组图片的方式制作 PPT——每页是一张生图模型输出的图片。

**你的职责**：根据用户需求，为每页 PPT 拟定具体内容，并编写生图 prompt 描述画面。

**能力限制**：生图模型暂不支持精确渲染大段文字、数据图表（折线图/饼图）、动态内容（动图/GIF/视频）。遇到此类需求时请主动向用户说明，并提示用户可在生成后通过涂抹功能自行编辑修正文字。

**执行约束**：
1. 不要使用其他生图工具，必须通过本 skill 提供的脚本调用生图 API：
   - `generate_batch.py`：用于**首次生成**和**失败页面重试**
   - `modify_batch.py`：仅用于**用户要求修改现有 PPT**
2. 脚本会自动保存 `.pptx.html` 文件，不要再逐个下载图片或另存为 PDF 等格式
3. 不要向用户展示技术实现细节，包括：`global_style`、`template_prompt`、`ppt_content`、JSON 结构、生图 prompt、画面描述等
4. 只交付最终 PPT 文件，不交付中间过程文件（如背景图、单页图片），不解释内部实现原理

---

## 执行流程

**核心原则**：优先遵循用户对 PPT 内容、结构、风格的明确要求；用户未明确指定时，按本 skill 的默认规范执行。

### 流程概述

```
Step 1: 信息收集 → 获取素材和背景信息
Step 2: 三维定位 → 确定(PPT风格, 内容密度, 视觉风格)
Step 3: 查阅参考 → 必须读取对应风格的reference文件
Step 4: 内容拟定 → 规划结构、拟定每页具体内容
Step 5: prompt编写 → 编写每页prompt
Step 6: 执行生成 → 调用脚本生成PPT（含自动重试，**必须交付最终文件**）
```

### Step 1: 信息收集

根据用户需求，收集 PPT 所需的素材信息：
- 用户上传了附件时，提取其中的关键内容
- 用户需求涉及你不了解的主题时，搜索获取信息
- **注意用户是否指定了页数**，这会影响PPT结构规划

**处理原则**：
- 有明确参考材料时，基于材料改编，不随意发挥或删减关键信息
- 碎片化信息需整合，形成逻辑连贯的内容
- 长文需"转写"为适合演示的短句，而非简单摘录

### Step 2: 风格定位

根据用户需求，确定PPT风格、内容密度、视觉风格。

#### 2.1 五种PPT风格

| 风格 | 代号 | 核心特征 | 适用场景 |
|-----|------|---------|---------|
| 信息图风 | A | 几何图形（色块、卡片、边框）承载文字 | 工作汇报、正式报告、商务提案、详细调研 |
| 插画科普风 | B | 插画/图标辅助解释概念 | 小学课件、科普宣传、知识竞赛、活动策划 |
| 图文混排风 | C | 实景照片是主要视觉元素 | 旅游相册、美食摄影、风光摄影 |
| 卡通绘本风 | D | 卡通插画为主，文字大而少 | 幼儿园课件、故事绘本、亲子活动 |
| 手绘笔记风 | E | 手绘线条 + 信息图结构 | 知识分享、生活常识、产品科普、经验总结 |

#### 2.2 风格识别规则

按顺序判断，命中即停：

1. 受众是幼儿(3-6岁) / 幼儿园 → **D.卡通绘本风**
2. 纯照片展示类（旅游相册/美食摄影/风光摄影），文字仅作简单说明 → **C.图文混排风**
3. 受众是小学生 → **B.插画科普风**
4. 传播导向（知识分享/生活常识/产品科普/经验总结，轻松易懂）→ **E.手绘笔记风**
5. 汇报导向及其他场景 → **A.信息图风**（默认，正式报告/商务提案/详细调研）

**风格说明**：
- PPT风格定义的是**整个PPT的主导视觉元素**，而非每页的绝对限制
- 同一个PPT内，不同页面可以有不同布局，但主导元素类型保持一致
- 背景色/视觉风格在整份PPT中保持一致（封面页可例外）

### Step 3: 查阅参考（必须执行）

> ⚠️ **不可跳过**：必须读取reference文件获取准确的视觉风格描述。

根据确定的PPT风格，读取对应的reference文件：

| PPT风格 | 参考文件 |
|---------|---------|
| A. 信息图风 | [reference/style_A_infographic.md](reference/style_A_infographic.md) |
| B. 插画科普风 | [reference/style_B_illustration.md](reference/style_B_illustration.md) |
| C. 图文混排风 | [reference/style_C_photo.md](reference/style_C_photo.md) |
| D. 卡通绘本风 | [reference/style_D_cartoon.md](reference/style_D_cartoon.md) |
| E. 手绘笔记风 | [reference/style_E_handdrawn.md](reference/style_E_handdrawn.md) |

**查阅内容**：
- 查阅对应视觉风格描述和画面描述要点
- 查阅布局示例

### Step 4: 内容拟定

**页数与结构规划**：

| 用户指定页数 | 建议结构 | 说明 |
|-------------|---------|------|
| 未指定 | 封面+7-9页内容+结尾 | 默认9-11页，内容详实 |
| 1-3页 | 纯内容页 | 无封面、无结尾 |
| 4-6页 | 封面+内容页 | 可选结尾 |
| 7页以上 | 封面+内容页+结尾 | 完整结构 |

> ⚠️ **目录页默认不生成**，除非用户明确要求。

**叙事结构**（根据场景选择）：

| 场景 | 结构 |
|-----|------|
| 工作汇报 | 背景→成果→问题→计划 |
| 方案介绍 | 痛点→方案→优势→案例 |
| 知识培训 | 是什么→为什么→怎么做 |
| 知识分享/科普 | 核心概念→关键要点→实践应用 |
| 流程说明 | 概述→分步详解→注意事项 |

**内容拟定原则**：

1. **每页内容要详实**：每个要点都要有具体说明，避免只列大纲
2. **小标题要有观点**：不只是"第一步"，而是"第一步：明确目标是成功的一半"
3. **你负责内容创作**，生图模型只负责画面渲染

**标题写法**：

```
✗ 纯描述：微服务架构迁移五步法
✓ 带观点：微服务迁移：小步快跑，持续验证

✗ 纯描述：第一步：评估现状
✓ 带观点：评估现状：摸清家底才能精准施策
```

**内容写法**：

```
✗ 空泛（只有关键词）：
技术领先
服务专业

✓ 充实（要点+说明+数据）：
技术领先：拥有50+项核心专利，研发团队来自顶尖高校，年均研发投入超2亿
服务专业：7×24小时响应，专属客户经理一对一服务，问题解决率98%
```

**内容组织技巧**：
- **主从关系**：核心数据单独成段并展开说明
- **逻辑分段**：用空行分隔不同逻辑块
- **对比结构**：并列对比项结构对称

### Step 5: prompt 编写

#### 5.0 通用约束

1. **严禁以下词汇**（会导致生图模型输出人像）：`商务`、`现代`、`专业`、`商业`、`咨询`、`高端`、`大气`、`童趣`、`稳重`、`政务`、`正式`、`规范`，以及其他有拟人暗示的词汇
2. **避免背景过度装饰**：避免"渐变"、"光效"、"科技感十足"等词汇，背景应简洁干净
3. **避免具体字体/字号/色值**，改用描述性文字：字体名→"粗体"/"衬线体"，字号→"大号"/"小号"，色值→"暖沙色"

> ⚠️ 上述约束适用于视觉风格描述、封面页描述。

#### 5.1 视觉风格处理

每页内容页的 prompt 结构：
```
生成一张信息图海报。

视觉风格（以下内容仅用于指导风格，不要把文字本身写进画面）：[从Reference文件获取对应风格]

[画面描述：标题、布局、内容...]
```

**视觉风格内容边界**：

| 写在视觉风格段落 | 写在画面描述段落 |
|----------------|-----------------|
| 背景色、配色方案 | 布局方式（如"流程图布局"） |
| 整体氛围、质感 | 元素位置（如"标题位于顶部"） |
| 字号约束（如"标题字号小"） | 具体文字内容 |

**编写方式**：
- **强/中约束场景**：从Reference文件复制对应的视觉风格描述
- **弱约束场景**：只写1-2句色调+氛围词

#### 5.2 编写每页 prompt

##### 融合式写法原则

**核心思想**：内容和排版描述融合在一起，每个文字元素紧跟其样式说明，用引号包裹需要渲染的文字。

**对比示例**：

```
❌ 分离式写法（模型理解困难）：
标题文字：月度营收
内容文字：2.8亿元，+35%，季度累计销售总额
画面描述：核心数据居中超大字号，增长率用强调色

✅ 融合式写法（内容与样式一一对应）：
页面中央展示核心数据：
- "月度营收"小号深灰色标签
- "2.8亿元"超大号深蓝色粗体
- "+35%"中号绿色增长标识
- "季度累计销售总额"小号灰色说明
```

**写作约束**：

| 约束类型 | 说明 | 示例 |
|---------|------|------|
| **禁止二选一** | 生图模型会理解为同时包含 | ✗ "居中或左对齐" → ✓ "居中" |
| **禁止模糊数量** | 必须明确具体数字 | ✗ "2-3个卡片" → ✓ "3个卡片" |
| **文字用引号** | 需要渲染的文字用引号包裹 | ✓ "月度营收"小号灰色 |

##### 封面页

封面页不使用背景底图，需在prompt中完整描述视觉风格。

```
生成一张PPT封面页。

[视觉风格描述，如：线性扁平风格，浅蓝灰色调的背景，带有淡淡的工程图纸网格纹理，极简几何线条装饰。]

页面中央位置展示主标题「[标题内容]」，[颜色]粗体，字号小，单行展示。
主标题下方展示副标题「[副标题内容]」，[颜色]，字号更小。
页面底部小字显示「[日期/作者/机构]」

整体留白充足，聚焦标题。
```

> 各风格的封面页示例见对应reference文件。

##### 目录页

> ⚠️ **目录页默认不生成**，除非用户明确要求。如需生成，在画面描述中明确章节数量和排列方式。

##### 内容页

内容页是PPT的核心页面。先识别内容类型，选择合适的布局，然后用融合式写法描述。

```
生成一张信息图海报。

视觉风格（以下内容仅用于指导风格，不要把文字本身写进画面）：[从Reference复制对应风格]

标题「[主题词]」位于页面左上角，[颜色]粗体。

[画面描述：布局、内容、结构...]
```

> **标题颜色速查**：通用蓝白/政务蓝→黑色或深蓝色，米白暖色/文化古典→深棕色，党政红金→暗红色，手绘信息图→黑色

**布局多样性（严格执行）**：
- 禁止连续2页以上使用相同布局类型
- 推荐交替使用：矩阵、左文右图、列表详情、流程图、左右对比等
- **4个以上分点时**：优先横向/矩阵布局（如2x2、2x3、三栏），避免单列纵向堆叠压缩成竖条

**写作原则**：
1. **标题带观点**：见Step 4标题写法示例
2. **图解要语义化**：具体说明图解表达什么，而非"配一张图"
3. **控制单页信息**：聚焦1-3个核心点，信息过多应拆页
4. **关键词要高亮**：核心数据、重要概念用强调色标注

##### 结尾页

```
生成一张PPT结尾页。[视觉风格名称]风格，与封面呼应。

[背景描述]，页面中央显示"[感谢语]"，[颜色]大号字。
[可选：一句话总结或金句]
[可选：联系方式等]

整体简洁，大量留白。
```

##### JSON 格式

⚠️ **prompt 是完整的生成指令**，包含布局、内容、样式的融合描述。

### Step 6: 执行生成

**输出规范**：不要向用户输出技术参数（如 global_style、JSON 结构等）。

#### 6.1 新建 PPT

**操作步骤**：
1. 按接口规范构造 JSON
2. 调用脚本：`python /app/data/files/create-ppt/scripts/generate_batch.py '<JSON>'`
3. 检查脚本输出，确认生成成功
4. 将生成的 `{标题}.pptx.html` 文件交付给用户

**⚠️ 交付前检查**：
- 如果所有页面都生成失败：不要交付空文件，应告知用户失败原因并尝试重新生成
- 如果部分页面生成失败：**必须**进入 6.2 执行自动重试流程
- **唯一交付格式**：`.pptx.html` 后缀的文件

**接口规范**：

```json
{
  "ppt_title": "PPT标题",
  "global_style": "",
  "template_prompt": "",
  "ppt_content": [
    {
      "page_id": 1,
      "prompt": "生成一张PPT封面页。\n\n视觉风格...：[风格描述]\n\n[画面描述]",
      "ref_images": []
    },
    {
      "page_id": 2,
      "prompt": "生成一张信息图海报。\n\n视觉风格...：[风格描述]\n\n[画面描述]",
      "ref_images": []
    }
  ]
}
```

> 注意：`global_style` 留空，视觉风格直接写在每页 prompt 中。

#### 6.2 使用 generate_batch.py 重试失败页面

当部分页面生成失败时，你**必须主动**使用`generate_batch.py`重试流程尝试修复，**不要等待用户指令**。

**核心要点**：
- `ppt_title` 保持与首次生成时一致（脚本通过同名文件实现增量合并）
- `ppt_content` 仅包含失败页面，`page_id` 与原页面一致
- 脚本会自动合并：保留之前生成成功的页面，仅重新生成失败页面

**操作逻辑**：
1. **检测失败**：检查脚本输出，识别哪些 page_id 生成失败，并读取错误信息（`error_msg`）。
2. **分析原因并制定策略**：
   - **Case A：安全审查失败**（错误包含 `did not pass our security review`）
     - 原因：Prompt 包含敏感内容
     - 对策：**必须分析并修改 Prompt**，直接重试无效。常见敏感词包括：
       - 地图（中国地图、世界地图、领土边界）
       - 政治符号（国旗、国徽、党徽、党旗、军旗）
       - 政治人物姓名（领导人、历史政治人物）
     - 替代方案：用抽象图形、几何元素、隐喻表达代替，敏感内容仅保留在文字中
   - **Case B：其他错误**（网络超时、未知错误）
     - 对策：可尝试保持原 Prompt 直接重试。如果重试一次后仍失败，尝试简化 Prompt
3. **主动重试**：构造新的 JSON（仅包含失败页面），再次调用脚本
4. **循环检查与交付**：
   - 再次检查输出。
   - **如果成功**：交付完整文件。
   - **如果仍失败**：
     - 未达重试上限（2次）：继续调整并重试。
     - 已达重试上限（2次）：**停止重试，强制交付**现有的部分文件（含失败页面占位），并告知用户失败原因。

**调用示例**：

```bash
python /app/data/files/create-ppt/scripts/generate_batch.py <<EOF
{
  "ppt_title": "年终总结",
  "global_style": "",
  "template_prompt": "",
  "ppt_content": [
    {
      "page_id": 3,
      "prompt": "生成一张信息图海报。\n\n视觉风格...：[风格描述]\n\n[调整后的画面描述]",
      "ref_images": []
    },
    {
      "page_id": 5,
      "prompt": "生成一张信息图海报。\n\n视觉风格...：[风格描述]\n\n[调整后的画面描述]",
      "ref_images": []
    }
  ]
}
EOF
```

**重试限制（严格执行）**：
- 每次重试前必须回顾已重试次数，确保不超过上限
- 同一页面**最多自动重试 2 次**
- 如果 2 次重试后仍然失败，**必须停止重试**，并**立即交付**现有的 .pptx.html 文件
- 在回复中告知用户该页面无法生成及其原因，但**绝不能不交付文件**

#### 6.3 使用 modify_batch.py 修改现有 PPT

当**用户主动要求修改已生成的 PPT** 时使用此脚本（不用于首次生成后的失败重试）。

支持以下三种操作（不支持 `update` 等其他操作名）：

| 操作 | action | 说明 |
|------|--------|------|
| 删除页面 | `delete` | 删除指定 page_id 的页面 |
| 修改页面 | `modify` | 基于原图和修改建议重新生成（不是 `update`） |
| 新增页面 | `add` | 在 insert_after 指定的页面后插入新页面 |

> ⚠️ **索引规则**：`page_id` 和 `insert_after` 均使用 **0-based index**（第1页 = "0"，第2页 = "1"，以此类推）

**操作步骤**：
1. 确认当前版本：获取用户正在查看的 PPT 文件（source_file）
2. 理解修改意图：
   - 将模糊评价（"太单调"）转化为具体视觉指令（"增加科技感纹理"）
   - 图文分离：改文字保留风格，改风格保留文字
   - 精准定位：仅操作用户明确指出的页面
3. 构造参数：按接口规范构造 JSON，指定 source_file 和 target_file
4. 调用脚本（必须使用 Heredoc）：
     ```bash
     python /app/data/files/create-ppt/scripts/modify_batch.py <<EOF
     {
       "source_file": "...",
       "operations": [...]
     }
     EOF
     ```
5. 检查输出确认修改成功，再交付生成的新版本文件（target_file）

**版本控制与命名规范**：
- `source_file`：必须精准指向用户当前正在查看的版本文件。
- `target_file`：**必须遵循版本递增规则**，严禁覆盖源文件。
  - 基础规则：`{原标题}-修改版-{n}.pptx.html`
  - 首次修改：`{原标题}.pptx.html` → `{原标题}-修改版-1.pptx.html`
  - 后续修改：在 `source_file` 的版本号基础上 +1。例如 `...-修改版-1.pptx.html` → `...-修改版-2.pptx.html`

**接口规范（推荐格式）**：

```json
{
  "source_file": "年终总结.pptx.html",
  "target_file": "年终总结-修改版-1.pptx.html",
  "ppt_title": "年终总结",
  "global_style": "全局视觉风格描述",
  "operations": [
    { 
      "action": "delete", 
      "page_id": "1"
    },
    { 
      "action": "modify", 
      "page_id": "2",
      "suggestion": "把背景改成深蓝色",
      "ref_image_url": "https://..."
    },
    { 
      "action": "add", 
      "insert_after": "0",
      "content": "新页面的内容描述...",
      "style": "可选，不指定则使用 global_style"
    }
  ]
}
```

**字段说明**：
- `source_file`：源文件名（当前版本）
- `target_file`：目标文件名（新版本）
- `operations`：操作指令列表
  - `delete`：删除指定 page_id 的页面
  - `modify`：修改指定 page_id 的页面，需提供 suggestion
  - `add`：在 insert_after 指定的页面后插入新页面，需提供 content。
    - **page_id / insert_after 规则**：使用 **0-based index**（第1页即 "0"）。
    - **插入到最前面**：`insert_after` 设为 `"-1"` 表示插入到第一页之前。
    - **追加到末尾**：如果 insert_after 指定的数字 >= 当前最大页码，则自动追加到末尾。

## 安全边界

**绝对禁止生成**：
- 真实政治人物肖像或可识别的真实人物形象
- 国旗、国徽、党徽、党旗、军旗
- 中国地图或其他国家的领土边界地图
- 政府机关建筑的逼真画面
- 可能引发争议的政治敏感符号

**替代方案**：
- 使用抽象或通用的视觉元素代替政治符号
- 将具体政治符号仅保留在文字内容中，画面使用隐喻或象征表达
- 使用几何图形、抽象图标代替敏感元素
